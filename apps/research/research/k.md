# 채용 시장의 변화

| 2021-2022 (황금기)      | 2025 ~ 현재 (빙하기) |

# 왜 이렇게 됐을까?

## 기업의 계산법이 바뀌었습니다

**과거:** "3년 후 성장 포텐셜에 베팅" → 신입 = 미래에 대한 투자

**현재:**

- 투자 시장이 얼어붙으며 **교육 여력 자체가 사라짐**
- AI가 주니어 업무를 대체하기 시작
- 신입 채용 = **비용 대비 효용 재계산**

> **"과거엔 3년 후를 봤다면, 지금은 3개월 안에 성과를 봐야 합니다."**

---

# 또 하나의 변수: AI

## 과거에 신입이 하던 일들

- **개발자:** 간단한 API 연동, 문서화, 기본 CRUD
- **기획자:** 경쟁사 분석, 기획서 초안

## 지금은?

- 개발자: **시니어 1명 + AI**가 더 빠르게 처리
- 기획자: **ChatGPT**가 기획서 초안을 10분 만에 작성

> **"시키는 일만 하는 주니어의 자리가 사라지고 있습니다."**

---

# 기업이 원하는 것

## 셀프 온보딩이 가능한 사람

- 문서를 보고 **스스로 파악하려고 시도**하는 사람
- 막히면 **먼저 찾아보고** 질문하는 사람
- 같은 실수를 **반복하지 않는** 사람
- **기초는 갖추고 온** 사람

## Before → After

# 탈락 패턴 #2: 남의 생각 복붙

## 개발자: 클론코딩

## 기획자: 벤치마킹

- "쿠팡 분석 → 우리 앱에 적용"
- **남의 서비스 캡처 + "이렇게 하면 좋겠다"**

# 왜 문제인가?

## 면접관의 시선

- 이 지원자만이 가진 고유의 색깔은 무엇일까?
- 진짜 고민해본걸까? 아니면 무엇을 참고만 한걸까?

**진짜 문제는 "왜"가 없다는 것**

- 넷플릭스가 **왜** 이 UI를 선택했는지 모르고 **뭘** 따라함
- 토스가 **왜** 이 플로우를 만들었는지 모르고 **겉모습**만 분석

---

# 넘어서는 법

## 클론/벤치마킹 + 알파

**개발자:**

- "넷플릭스 클론인데, **접근성(a11y)** 을 WCAG 2.1 기준으로 개선"
- "배민 클론인데, **오프라인 지원** 을 위해 PWA로 구현"

**기획자:**

- "토스 송금 플로우를 분석하고, **우리 타겟(60대)에 맞게** 재설계"
- "실제로 **5명에게 프로토타입 테스트** 진행"

> **"봤다"가 아니라 "해봤다"**

---

# 가장 좋은 건

## 나만의 문제 해결

- **개발자:** 동아리 회비 관리가 불편해서 만든 웹앱
- **기획자:** 우리 학교 수강신청이 불편해서 만든 개선안

> **내가 진짜 불편했던 것을 해결하는 서비스**

# 탈락 패턴 #3: 확인 가능한 포트폴리오

## 깃허브/노션 링크를 눌렀더니

- README가 없음 / 권한이 없어서 못 들어감
- 커밋 메시지, PR 등이 의미가 없음

- **접근 불가능한 포트폴리오는 없는 것보다 나쁩니다**
- 서류/면접 검토 시에 생각보다 많이 접속해봅니다.

# 넓고 얕은 vs 좁고 깊은

## 면접에서 실제로 일어나는 일

- 개발자: "React 쓴다고 하셨는데, Virtual DOM이 왜 필요한가요?"
- 기획자: "애자일로 일했다 하셨는데, 스프린트 회고 어떻게 진행하셨어요?"

> **하나만 깊게 물으면 대부분 막힙니다**

---

# 깊이란 무엇인가

## How(방법)가 아닌 Why(원리)

**얕은:** "useEffect에서 의존성 배열에 값 넣으면 그때 실행돼요"

**깊은:** "React는 렌더링마다 새 함수를 만드는데, 의존성 배열은 Object.is로 비교해서 실행 여부를 결정합니다. 클로저 때문에 stale closure 문제가 생길 수 있고..."

---

# 깊이를 만드는 방법

## 글쓰기

- 어떤 개념을 설명하려면 **내가 정확히 이해해야 함**
- 왜 이런지, 어떻게 동작하는지 알아야 함
- **대충 넘어간 부분이 적나라하게 드러남**

> **글쓰기 = 강제 깊이 파기**

# 블로그가 쌓이면

- 기술 글이 검색에 노출됨

- 면접 전에 면접관이 미리 읽고 옴
- 포트폴리오 자체가 콘텐츠가 됨

> **기록은 복리로 돌아옵니다**

# 무엇을 기록해야 할까?

## 가장 강력한 콘텐츠

**개발자:** 트러블슈팅

- 에러 상황 → 원인 분석 → 시도한 것들 → 해결책 → 배운 점

**기획자:** 의사결정 과정

- 문제 정의 → 가설 → 검증 방법 → 결과 → 배운 점

> **의미 없는 기록 100개 < 치열한 고민 1개**

---

# 기록 습관 만들기

## 현실적인 방법

- **진짜 고민했던 것만** 기록
- 해결하면 30분 안에 바로 정리
- 완벽하게 쓰려고 하지 말고 일단 공개

> **쓰는 습관이 먼저, 퀄리티는 나중에**

# 제 대답: 네, 더 해야 합니다

## 이유 1: AI 결과물은 당신 실력이 아닙니다

- 개발자: "이 코드 왜 이렇게 짰어요?"
- 기획자: "이 기능 왜 이렇게 정의했어요?"

**"AI가 이렇게 해주더라고요"는 답이 아닙니다**

> **기본기 없으면 면접 통과 불가**

---

# 이유 2: AI의 한계 - Hallucination

## AI가 자신있게 틀린 답을 줍니다

- **개발자:** 존재하지 않는 라이브러리 추천, deprecated API 사용
- **기획자:** 틀린 시장 데이터, 존재하지 않는 사례 인용

**기초가 없으면 맞는지 틀린지 판단 불가 → 책임은 내가 짐**

---

# 그래서 기본기란?

## 개발자: 프레임워크보다 원리

- HTTP/HTTPS 동작 원리
- 브라우저 렌더링 파이프라인
- JavaScript 이벤트 루프

**React는 바뀌어도 HTTP는 안 바뀝니다**

## 기획자: 툴보다 사고방식

- 문제 정의 능력, 가설 수립과 검증, 데이터 해석

**Figma는 바뀌어도 "왜"를 묻는 건 안 바뀝니다**

# AI가 못하는 것

- **"왜"를 묻는 것:** AI는 주어진 것에 답할 뿐
- **맥락을 파악하는 것:** 우리 회사, 우리 사용자, 우리 상황
- **책임지는 것:** 결과에 대한 오너십

> **AI는 도구입니다. 방향을 정하는 건 사람입니다.**

---

# 기업의 AI 활용 현실

## 생각보다 제한이 많습니다

- **보안 규제:** 사내 데이터를 외부 AI에 입력 금지
- **승인된 툴만 사용:** ChatGPT, Copilot 등 개인 사용 제한
- **사내 AI 도입 초기:** 있어도 제대로 쓰는 사람이 드묾

> **취준 기간 = AI 활용 역량을 키울 수 있는 시간**

---

# 어필할 수 있는 포인트

## "AI 잘 씁니다"가 아니라

- **실제 문제 해결 사례:** "이 프로젝트에서 AI로 X를 자동화했습니다"
- **한계 인식:** "AI가 틀릴 수 있어서 검증 과정을 넣었습니다"

❌ "ChatGPT 많이 써봤습니다"
⭕ "AI 출력을 **검증하는 체크리스트**를 만들어 썼습니다"

> **도구를 쓰는 건 누구나 합니다. 잘 쓰는 건 다릅니다.**
