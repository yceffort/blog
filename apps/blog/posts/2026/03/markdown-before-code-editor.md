## TL;DR

- AI 시대의 개발자는 코드를 쓰는 사람이 아니라, 요구사항을 명세하고 결과를 검증하는 사람으로 역할이 이동합니다.
- 구현을 바로 시작하지 말고, 요구사항을 먼저 명세하고 → LLM에 구현을 맡기고 → 사람은 검증과 판단에 집중하는 방식으로 작업 흐름을 바꿔야 합니다.
- 동시에, 직접 코드를 작성하는 시간을 의도적으로 남겨서 판단 능력을 유지해야 합니다.

## AI가 대체하는 순서가 있다고 생각합니다

요즘 AI 코딩 도구를 쓰면서, 하나의 불편한 질문이 계속 머릿속에 남아 있습니다. "지금 내가 하고 있는 이 작업, 1년 뒤에도 사람이 해야 하는 일인가?" 아마 이 생각은 저만 하는 게 아닐 겁니다.

개발자의 업무를 단계와 난이도별로 러프하게 나열하면 이렇습니다.

**코드 작성 → 코드 리뷰 → 아키텍처 설계 → 기술 의사결정**

AI는 이 순서의 앞쪽부터 침범하고 있습니다.

코드 작성은 이미 AI가 사람보다 빠르고, 경우에 따라 더 정확합니다. 실제로 체감한 사례를 하나 들면, 예전에는 반나절은 잡아야 했던 컴포넌트 구현을 Claude Code에 명세를 넘겼더니 3분 안에 동작하는 코드가 나왔습니다. 타입도 맞고, 엣지 케이스 처리도 꽤 괜찮았습니다. 제가 직접 짠 것과 품질 차이가 거의 없었습니다.

코드 리뷰도 정적 분석 수준을 넘어서 맥락을 이해한 피드백이 가능해지고 있습니다. 여담인데 Claude Code Review가 PR 하나에 15달러라고 합니다. 아직은 사람 리뷰어가 더 싸긴 한데(??), 이 가격이 빠르게 떨어질 거라는 건 누구나 예상할 수 있습니다.

아키텍처 설계와 기술 의사결정은 아직 사람의 영역입니다. 근데 솔직히, 이것도 언제까지일지는 모르겠습니다.

그런데 우리 대부분은 아직 앞쪽 두 단계에서만 AI를 쓰고 있습니다. Copilot으로 코드를 자동완성하고, PR에 AI 리뷰어를 붙이는 정도. 정작 AI가 대체하기 어려운 뒤쪽 역량 — 요구사항을 구조화하는 능력, 도메인 지식 기반의 설계 판단 — 을 키우는 데는 시간을 쓰지 않습니다.

실제로 Claude Code 같은 도구가 도입된 이후로, 주변에서 관찰되는 패턴이 있습니다. 작업이 생기면 바로 LLM에 던지고, 결과물을 받아서 PR을 올리고, 끝. 이 루프가 점점 짧아지고 있습니다. 빠르니까 편하고, 편하니까 반복합니다. 문제는 이 루프 안에서 사람이 하는 일이 "던지기"와 "붙여넣기"뿐이라는 겁니다. 요구사항을 분석하는 시간도, 결과물을 깊이 검증하는 시간도 빠져 있습니다.

솔직하게 말하면, 개발자로서 우리의 수명은 이 네 단계 중 AI가 쉽게 대체하지 못하는 뒤쪽 역량에 달려 있습니다. 코드를 빠르게 타이핑하는 능력은 예전에도 차별화 요소가 아니었고, 지금은 더더욱 아닙니다. 요구사항을 정확하게 분석하고, 제약조건을 식별하고, 트레이드오프를 판단하는 능력. 이건 코드를 많이 짜는 것만으로는 늘지 않습니다.

그래서 작업 방식을 바꾸는 걸 제안합니다. 단, 이 방식은 프로덕션 장애 대응이나 핫픽스, 급한 디버깅에는 적용하지 않습니다. 새로운 기능 개발, 계획된 리팩토링, 컴포넌트 설계 — 이런 "계획 가능한 작업"에 대한 이야기입니다.

"요구사항 먼저 정리하고, 작게 쪼개고, 짧은 주기로 회고하라." 여기까지 읽으면 이런 생각이 들 수 있습니다 — 이거 그냥 애자일/스크럼 아닌가? 맞습니다. 골격은 같습니다. 다른 건 실행 계층입니다. 기존에는 요구사항을 정리한 뒤 사람이 직접 구현했습니다. 이제는 그 구현을 LLM에 위임하고, 사람은 명세의 정밀도와 결과물의 검증에 집중합니다. 프로세스가 바뀐 게 아니라, 프로세스 안에서 사람이 서 있는 위치가 바뀐 겁니다.

## 소규모 작업: 코드 에디터 대신 마크다운부터 열기

새로운 컴포넌트를 만들거나, 기능을 하나 추가하거나, 단순한 리팩토링을 하는 경우 — 지금까지는 바로 코드 에디터를 열고 구현부터 시작했습니다. 앞으로는 이렇게 바꿉니다.

**1단계: 마크다운으로 요구사항을 먼저 작성합니다.**

무엇을 만들어야 하는지, 제약조건은 무엇인지, 기대하는 결과물은 어떤 형태인지를 문서로 정리합니다. "무엇을"과 "왜"를 코드가 아닌 글로 먼저 정의하는 겁니다. 이때 허용 기준(acceptance criteria)도 같이 적습니다. 이 허용 기준이 나중에 검증의 기준이자 테스트 케이스가 됩니다.

여기서 정말 중요한 건, 이 명세 작성 자체를 AI에 맡기지 않는 겁니다. 요구사항을 정리하는 과정이 곧 설계이고, 그 설계를 사람이 직접 하는 것이 이 워크플로우의 핵심입니다. AI에게 "요구사항 정리해줘"라고 던지는 순간, 다시 "던지기 → 붙여넣기" 루프로 돌아갑니다. 물론 직접 쓴 초안을 AI에게 보여주고 빠진 부분이나 표현을 다듬는 정도는 괜찮습니다. 핵심은 첫 번째 생각이 사람에게서 나와야 한다는 겁니다. 이 단계의 시작부터 AI에게 외주화해버리면, 이후 모든 프로세스가 무의미해집니다.

**2단계: 이걸 LLM에 지시합니다.**

정리한 요구사항을 기반으로 LLM에 구현을 맡깁니다. 핵심은 "내가 타이핑하는 것"이 아니라 "내가 정의한 것을 실행시키는 것"입니다.

**3단계: 결과물을 사람이 직접 검증합니다.**

여기가 가장 중요합니다. LLM이 만든 코드가 요구사항을 충족하는지, 보안 취약점은 없는지, 엣지케이스는 처리되었는지, 성능에 문제가 없는지 — 이 검증은 반드시 사람이 해야 합니다.

기존에 코딩에 쏟던 고민의 밀도를, 이제는 리뷰에 쏟아야 한다는 뜻입니다. 리뷰까지 AI에 맡기면 어떻게 될까요? 같은 모델이 작성하고, 같은 모델이 검증하는 구조가 됩니다. 같은 맹점을 가진 시스템이 자기 자신을 검증하는 거니까, 놓치는 부분도 같습니다. 코드 작성에 들이던 시간을 절약한 만큼, 리뷰에 들이는 사람의 시간과 사고는 오히려 늘어야 합니다.

Before와 After를 비교하면 이렇습니다.

```
Before:
1. 코드 에디터 열기
2. 구현하면서 요구사항을 머릿속으로 정리
3. 다 짜고 나서 PR 올리기
4. 리뷰에서 "이거 왜 이렇게 했어요?" 질문 받기

After:
1. 마크다운으로 요구사항, 제약조건, 수용 기준 작성
2. LLM에 지시
3. 결과물을 수용 기준 기반으로 사람이 직접 검증
4. 요구사항 명세를 코드와 함께 기록으로 남김
```

이렇게 하면 부수효과도 있습니다. 요구사항을 글로 먼저 쓰면 구현 전에 설계를 강제하게 되고, "일단 짜고 보자"가 아니라 "뭘 짜야 하는지 먼저 정의하자"가 기본 모드가 됩니다. 그리고 이 명세는 코드와 함께 남깁니다. 코드만 있는 저장소에서는 "왜 이렇게 만들었는지"를 알 수 없지만, 요구사항 명세가 함께 있으면 코드의 의도가 기록으로 남습니다. 나중에 다른 사람이 — 혹은 미래의 내가 — 이 코드를 수정할 때, PR을 뒤지지 않아도 맥락을 바로 파악할 수 있습니다.

한 가지 더. 직급이 올라갈수록 코드보다 글을 쓸 일이 많아집니다. 기술 의사결정을 보고서로 정리하거나, 기획/디자인을 설득하기 위해 근거를 구조화하는 일. 요구사항 명세를 반복적으로 작성하는 과정이 이 능력을 자연스럽게 키워줍니다.

## 대규모 작업: 설계 시점부터 함께 분석하고 쪼개기

리팩토링이나 규모가 큰 요구사항은 다릅니다. 혼자 마크다운 하나 쓰고 LLM에 던질 수 있는 크기가 아닙니다.

**바로 착수하지 않습니다.** 2인 이상이 설계 시점부터 참여해서 전체 요구사항을 분석합니다. 이 분석 결과를 마크다운으로 정리하되, "무엇을 만들 것인가"뿐 아니라 "무엇을 만들지 않을 것인가"도 명시합니다. 스코프가 불명확한 대규모 작업은 대부분 여기서 문제가 시작되기 때문입니다.

이 단계에서는 도메인 지식과 코드베이스에 대한 이해가 전제됩니다. "상태 관리를 어떻게 할지", "렌더링 전략을 SSR로 갈지 CSR로 갈지", "에러 바운더리를 어디에 둘지" — 이런 판단 없이 마크다운만 적으면 LLM이 내놓는 결과물의 품질을 판단할 수 없습니다. 비즈니스 도메인을 모르면 요구사항 자체를 제대로 정의할 수 없고, 코드베이스를 모르면 기존 구조와 충돌하는 결과물을 걸러낼 수 없습니다. 코드를 안 짜는 게 아니라, 코드를 짤 수 있는 사람이 안 짜는 겁니다.

**정리된 전체 요구사항을 작은 단위로 분해합니다.** "작은 단위"의 기준은 명확합니다: 하나의 마크다운 명세로 충분히 설명 가능한 크기, 그리고 하나의 PR로 리뷰 가능한 크기. 이렇게 쪼개면 각각의 작은 요구사항이 앞서 말한 소규모 작업의 흐름을 탑니다. 즉, 큰 문제를 "마크다운 명세 가능한 크기"로 만드는 것이 핵심입니다.

```
대규모 요구사항
  → 2인 이상 설계 분석 + 마크다운 정리
    → 작은 요구사항 A (→ 소규모 작업 흐름)
    → 작은 요구사항 B (→ 소규모 작업 흐름)
    → 작은 요구사항 C (→ 소규모 작업 흐름)
    → ...
```

## 짧은 피드백 루프: 1주 단위 회고

LLM 기반 작업은 기존보다 속도가 빠릅니다. 속도가 빠르면 방향이 틀어지는 것도 빠릅니다. 그리고 무엇보다 한 번도 해보지 못한 방식이라 시행착오도 많을 겁니다. 그래서 회고 주기를 1주로 짧게 가져갑니다.

1주 단위로 확인할 것은 네 가지입니다:

- 지난주 작성한 요구사항 명세의 품질은 어땠는가. LLM에 지시했을 때 기대한 결과가 나왔는가, 아니면 명세가 부족해서 수정이 많았는가.
- 분해한 작업 단위가 적절했는가. 너무 컸거나, 너무 잘게 쪼개진 건 아닌가.
- 검증 과정에서 놓친 것은 없었는가.
- LLM과 작업하면서 얻은 레슨런은 무엇인가.

2주, 4주로 늘리면 피드백이 느려져서 같은 실수를 반복하게 됩니다. 1주면 빠르게 교정할 수 있습니다.

## 이 방식이 잘 안 될 수 있는 조건들

솔직히 이 방식에 뻔한 한계가 있습니다. 인지하고 시작하는 것과 모르고 시작하는 건 다르니까 미리 꺼내놓겠습니다.

**명세 작성 능력의 편차.** 코드는 돌려보면 되는데, 명세는 품질을 즉시 측정하기 어렵습니다. 같은 요구사항을 두 사람이 쓰면 LLM 아웃풋 품질이 크게 달라질 겁니다. 앞서 말한 1주 회고에서 "명세 품질 확인"이 사실상 이 문제에 대한 답입니다. 명세를 쓰고, 결과를 보고, 다음에 더 잘 쓰는 사이클을 팀 단위로 돌리는 수밖에 없습니다.

**"명세 쓰는 시간 = 비효율"로 보이는 문제.** 빠른 딜리버리를 요구받는 환경에서, 마크다운부터 여는 사람은 당장 코드 치는 사람보다 느려 보입니다. 이건 팀 문화 차원에서 합의가 없으면 개인이 아무리 해봐야 지속이 안 됩니다. "명세를 먼저 쓰는 건 느린 게 아니라 설계를 앞당기는 것"이라는 합의가 팀 차원에서 있어야 합니다.

**검증 능력의 점진적 퇴화.** 바로 다음 섹션에서 "신성불가침의 시간"으로 이걸 방어하자고 제안하는데, 솔직히 이건 자기 규율에 의존하는 해법이라 구조적으로 약합니다. 바빠지면 가장 먼저 잘리는 시간이 "직접 짜는 시간"일 겁니다. 이 방법의 가장 큰 리스크가 바로 이겁니다. 구조적 해법은 아직 없습니다. 그래도 의식적으로 남겨두는 것과 아예 생각하지 않는 것 사이에는 차이가 있다고 봅니다.

## 신성불가침의 시간: 직접 짜는 영역을 남겨두기

여기까지 읽으면 "그럼 사람은 이제 코드를 안 짜는 건가?" 싶을 수 있는데, 그렇지 않습니다. 1주일의 작업 중 일부는 "직접 짜는 시간"으로 의식적으로 남겨둘 필요가 여전히 있습니다. 기술적 도전이 필요한 영역이나 내가 부족하다고 느끼는 영역 — 이 시간만큼은 LLM에 맡기지 않는 신성불가침의 영역으로 둡니다. 내가 먼저 작성해보고, 그 다음에 AI에게 의견을 구합니다. 순서가 반대가 되면 학습 효과가 사라집니다.

특히 경험이 적은 개발자일수록 이 시간의 비중을 의식적으로 높여야 합니다. 디버깅에서 직접 가설을 세우고 부딪혀본 횟수가, 나중에 AI 결과물을 보고 "이건 아닌데"라고 판단할 수 있는 근거가 됩니다. AI에 먼저 물어보는 습관이 붙으면 그 근거 자체가 쌓이지 않습니다. "AI가 짜준 코드를 읽었을 뿐인데 나도 짤 수 있을 것 같은" 감각을 항상 경계해야 합니다. 그건 이해가 아니라 익숙함입니다. AI에게 물어보기 전에 10분이라도 먼저 자기 가설을 세워보는 것, 그게 익숙함과 이해 사이의 간극을 메우는 가장 현실적인 방법입니다.

왜 이렇게까지 해야 할까요?

우선, 개발자로서의 즐거움은 여전히 직접 만드는 데서 나올 때가 있습니다. 코드를 직접 짜면서 느끼는 몰입감, 동작하는 결과물을 만들어냈을 때의 만족감 — 이걸 전부 LLM에 넘겨버리면 개발이라는 일 자체의 동기가 사라집니다. 효율만 남고 재미가 없는 일을 오래 할 수 있는 사람은 많지 않습니다.

또한, 직접 프로파일러를 열고 가설을 세우고 하나씩 시도해본 사람만이 다음에 처음 보는 문제 앞에서 자기 가설을 세울 수 있습니다. AI한테 "이 컴포넌트 렌더링 최적화해줘"라고 시키면 `useMemo` 넣고 `React.memo` 감싸고 결과물을 주겠지만, 그걸 읽는 것과 직접 삽질한 것 사이에는 결정적인 차이가 있습니다. 패턴이 안 맞는 상황에서 스스로 판단할 수 있는가. AI 딸깍 학습은 "아는 것"을 늘려주지만, 직접 구현하는 학습은 "판단할 수 있는 것"을 늘려줍니다. 그리고 최종적인 역량의 차이는 "판단할 수 있는 것"에서 갈립니다.

그리고 무엇보다, 코드를 아예 안 짜는 사람의 검증 능력은 시간이 지나면서 같이 퇴화합니다. 앞에서 제안한 워크플로우의 핵심이 "판단과 검증"인 이상, 직접 짜보는 경험을 놓으면 그 핵심이 무너집니다.

## 표현하는 수단이 바뀌는 것뿐입니다

코드 타이핑의 비중이 줄고, 요구사항 명세와 검증의 비중이 늘어나는 건 이미 일어나고 있는 현실입니다. 이 흐름을 받아들이되, "마크다운 잘 쓰면 된다"로 단순화하고 싶지는 않습니다. 본질은 여전히 소프트웨어 엔지니어링 역량이고, 그 역량을 표현하는 매체가 코드에서 명세로 일정 부분 이동하는 것뿐입니다.

앞으로 어떻게 될지는 솔직히 아무도 모릅니다. 하지만 막연히 불안해하는 건 아무것도 바꾸지 않습니다. 서두에서 말한 것처럼, 우리의 수명은 AI가 대체하기 어려운 역량에 달려 있습니다. "아는 것"이 아니라 "판단할 수 있는 것"을 늘려야 합니다. 코드든 명세든, 결국 중요한 건 소프트웨어가 풀어야 할 문제를 정확히 정의하고, 그 결과물이 정말 문제를 풀었는지 검증할 수 있는 능력입니다.

그 능력은 도구가 바뀌어도 유효합니다.
